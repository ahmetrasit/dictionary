# V2 Production Runbook

This is the production path for Quran-corpus root campaigns. It produces one
current minimal-input master entry and three deterministic consumer projections
per root envelope and target language. Run every command from the repository
root.

The production workflow uses one root-level writer and one evidence-bound
semantic reviewer per accepted response. The reviewer reports bounded issues
but never rewrites the entry. Deterministic scripts prepare, stage, validate,
accept, assemble, render, publish, and project.

## Prerequisites

- Python 3 and a controller runtime with bounded worker delegation are available.
- Canonical root packets exist under `data/output/root_packets/`.
- The deterministic inputs listed in `orchestration/entry-creation.spec.md`
  exist or can be generated by the preparer.
- The writer/reviewer model and reasoning profile are supplied by the user or
  current campaign configuration. Any explicit worker cap is an additional
  upper bound. Do not infer these settings from this runbook.
- Only the top-level controller launches workers through native delegation.
  Workers cannot spawn other workers, request `codex exec`, or run another
  agent-launching command.
- Root packet generation is not agent work. If packets are missing, generate
  them with deterministic coordinator commands before building the production
  queue; do not spend writer/reviewer agent slots on `scripts/root_packet.py`.

Protected-name policy is not a campaign blocker by itself. If
`v2/policy/protected_names/<root-envelope>.json` is absent, `create_entry.py`
may emit a coordinator fallback policy that classifies every lexical unit as
`ordinary`. A reviewed policy can replace that fallback when proper-name
classification matters, but the writer must never infer the policy.

## Root Queue

Build campaign queues from packet files, not guessed IDs:

```text
data/output/root_packets/root_*.json
```

Sort by numeric root envelope. Skip missing numeric IDs such as packet gaps, and
treat combined packet envelopes such as `root_000099--root_000100` as one root
workflow. Each root workflow includes every accepted branch in that packet.

Interpret "first N roots" as the first N sorted packet envelopes; a combined
envelope counts as one workflow. For "through/until root N", include envelopes
whose first numeric component is at most N and keep a combined envelope intact
at the boundary. An explicit envelope list means exactly those existing packet
files. Do not launch work for an ID without a packet.

The queue starts after packet generation has completed. A root workflow begins
with the controller running `create_entry.py`; it does not include a worker
whose only job is to create or refresh the packet.

## Single-Root Sequence

The controller runs all commands in this section from the repository root. It
does not delegate a command to a worker. Replace every angle-bracket placeholder
with a concrete value before execution; never pass or paste the placeholder
literally.

Prepare deterministic evidence and the canonical task:

```sh
python3 v2/scripts/create_entry.py <root-envelope> --language tr
```

If a canonical writer fragment exists, check it first and skip the writer only
on exit zero:

```sh
python3 v2/scripts/check_root_writer.py \
  v2/work/entry_creation/<root-envelope>/tr/tasks/root_writer.json \
  v2/work/entry_creation/<root-envelope>/tr/fragments/<root-envelope>_entry.json
```

Otherwise the controller stages the writer:

```sh
python3 v2/scripts/stage_root_writer.py v2/work/entry_creation/<root-envelope>/tr/tasks/root_writer.json
```

If the declared writer output already exists after staging, run
`validate_agent_output.py` before launching a worker. A valid current response
can be accepted directly. Preserve an invalid response and route its exact error
to the one writer continuation for that root.

When authored work is still required, the controller delegates one root writer,
not a script runner. Bind it to the staged `input/instructions.md` and
`output/<root-envelope>_entry.json`, forbid subagents, and retain that writer's
session identity through review and finalization. The writer may run only its
staged `validation.command`.

After the writer returns, the controller validates and accepts:

```sh
python3 v2/scripts/validate_agent_output.py v2/work/entry_creation/<root-envelope>/tr/input/task.json
python3 v2/scripts/accept_root_writer.py v2/work/entry_creation/<root-envelope>/tr/tasks/root_writer.json
```

Do not run acceptance after a failed validation. Send the exact error to the
same writer for an in-place correction, then rerun validation. Never edit or
copy the authored JSON manually.

Prepare the semantic review:

```sh
python3 v2/scripts/prepare_root_review.py \
  v2/work/entry_creation/<root-envelope>/tr/tasks/root_writer.json \
  v2/work/entry_creation/<root-envelope>/tr/fragments/<root-envelope>_entry.json
```

If a canonical review fragment exists, validate any stored verdict:

```sh
python3 v2/scripts/check_root_review.py \
  v2/work/entry_creation/<root-envelope>/tr/tasks/root_reviewer.json \
  v2/work/entry_creation/<root-envelope>/tr/fragments/root_review.json \
  --any-verdict
```

On exit zero, route the stored `pass`, `repair`, or `editorial_review` verdict;
do not launch a duplicate reviewer for a valid non-pass. For a stored non-pass,
rerun `accept_root_review.py` with the canonical review fragment as both its
response and `--output` to regenerate missing deterministic error/scope sidecars.

```sh
python3 v2/scripts/accept_root_review.py \
  v2/work/entry_creation/<root-envelope>/tr/tasks/root_reviewer.json \
  v2/work/entry_creation/<root-envelope>/tr/fragments/root_review.json \
  --output v2/work/entry_creation/<root-envelope>/tr/fragments/root_review.json
```

On nonzero check exit, the fragment is stale or invalid. The controller then
stages the package, delegates one independent semantic reviewer, and validates
and accepts its output. First stage:

```sh
python3 v2/scripts/stage_root_reviewer.py v2/work/entry_creation/<root-envelope>/tr/tasks/root_reviewer.json
```

If a current staged review output exists, validate and accept it before launching
a duplicate reviewer. Otherwise delegate the reviewer with
`review/input/instructions.md -> review/output/root_review.json`. After it
returns, run:

```sh
python3 v2/scripts/validate_agent_output.py v2/work/entry_creation/<root-envelope>/tr/review/input/task.json
python3 v2/scripts/accept_root_review.py v2/work/entry_creation/<root-envelope>/tr/tasks/root_reviewer.json
```

The reviewer receives only the staged package and cannot contact the writer or
spawn subagents. After acceptance, inspect the stored verdict:

- `pass` proceeds to finalization;
- `repair` follows the bounded repair procedure below;
- `editorial_review` parks the root for user judgment.

An acceptance command succeeding does not itself mean the verdict was `pass`.

Finalize and publish only after the review passes:

```sh
python3 v2/scripts/finalize_entry.py <root-envelope> --language tr
python3 v2/scripts/validate_entry.py v2/entries/tr/<root-envelope>.json
python3 v2/scripts/render_entry.py v2/entries/tr/<root-envelope>.json --check
```

## Campaign Loop

Limit running worker turns to the runtime's currently available capacity and any
explicit lower campaign cap. Do not assume a fixed slot count. Top up as soon as
capacity opens.

One active workflow means one root envelope and target language. Do not run two
writers for the same root, do not start a reviewer before the writer has been
accepted, and do not start a repair while the same root has an in-flight review.
Different roots may run concurrently. Retain each writer's session identity
until its root publishes or parks because repair and surface-form queues return
to that writer; an idle session handle is not a reason to launch a replacement.
A reviewer can be released after its output is validated and accepted; a
repaired response gets a fresh independent reviewer.

The controller operates every root state machine itself. Do not spawn per-root
controllers, command runners, monitors, queue builders, validators, or
publishers.

A root publishes when writer validation, writer acceptance, semantic-review
pass, finalization, master validation, and Markdown `--check` all succeed. A
root parks when a prepare blocker, operational failure, semantic ambiguity, or
exhausted bounded repair prevents publication. Parked roots do not block the
campaign queue.

## Repair Routing

Writer or reviewer validation errors go back to the same active agent. The
agent keeps the actual output file, changes only the fields required by the
exact validator error, and reruns the staged validation command.

For semantic-review `repair`, accept the review so it writes
`review/output/semantic_review_error.txt` and
`review/output/repair_scope.json`, then restage the same writer with the
previous accepted fragment:

```sh
python3 v2/scripts/stage_root_writer.py \
  v2/work/entry_creation/<root-envelope>/tr/tasks/root_writer.json \
  --previous v2/work/entry_creation/<root-envelope>/tr/fragments/<root-envelope>_entry.json \
  --repair-error v2/work/entry_creation/<root-envelope>/tr/review/output/semantic_review_error.txt \
  --repair-scope v2/work/entry_creation/<root-envelope>/tr/review/output/repair_scope.json
```

Continue the retained writer. If orchestration resumed without its session
handle, launch one bounded writer continuation from the staged previous
response, exact error, and scope; do not create a new full candidate. Then have
the controller validate and accept with the same scope:

```sh
python3 v2/scripts/validate_agent_output.py \
  v2/work/entry_creation/<root-envelope>/tr/input/task.json
python3 v2/scripts/accept_root_writer.py \
  v2/work/entry_creation/<root-envelope>/tr/tasks/root_writer.json \
  --previous v2/work/entry_creation/<root-envelope>/tr/fragments/<root-envelope>_entry.json \
  --repair-scope v2/work/entry_creation/<root-envelope>/tr/review/output/repair_scope.json
```

Then prepare a fresh semantic review bound to that repaired fragment and
delegate a fresh independent reviewer. Campaign mode permits one semantic repair
and one rebound review. If the rebound review still returns `repair` or
`editorial_review`, park the root with the exact review output and continue.

Structural validation errors go back to the worker that owns the current
artifact. The worker edits its declared output; the controller reruns validation
and the appropriate acceptance command. Never manually synchronize
`output/` and `fragments/`, and never edit a canonical fragment to bypass an
acceptance check. After two controller-mediated validation continuations for the
same artifact, park the root rather than looping indefinitely.

If finalization creates `inputs/transliteration_review.json` or
`inputs/name_review.json`, return only those generated queue files to the same
writer assignment. Use the retained session when available; after process
resumption, launch one bounded continuation authorized only for the exact queue
paths. The writer approves values in the queue and does not rewrite the accepted
entry response or run finalization. Then the controller reruns
`finalize_entry.py`. Transliteration and name queues may appear on separate
finalization passes. Do not route these routine queue values to the user or
create a new entry candidate. Park after two failed correction continuations for
the same queue.

## Known Prepare Blockers

Park the root and continue the campaign when preparation reports:

- no canonical root packet;
- missing source references in the branch dictionary roster;
- a branch with no lexical source claim;
- attachment grammar containing corpus-wide count claims.

These are input or evidence issues, not reasons to spawn replacement writers.

## Projections

After the master validates, projections are deterministic controller work. Run
them directly; do not delegate them:

```sh
python3 v2/scripts/project_entry.py v2/entries/tr/<root-envelope>.json \
  --projection translation_agent \
  --output v2/work/entry_creation/<root-envelope>/tr/verification/<root-envelope>.translation.json
python3 v2/scripts/project_entry.py v2/entries/tr/<root-envelope>.json \
  --projection user_dictionary \
  --output v2/work/entry_creation/<root-envelope>/tr/verification/<root-envelope>.user.json
python3 v2/scripts/project_entry.py v2/entries/tr/<root-envelope>.json \
  --projection scholar_view \
  --output v2/work/entry_creation/<root-envelope>/tr/verification/<root-envelope>.scholar.json
```

Quran-SLM remains an optional nomination lane. Do not relabel current QNet
discovery scores as Quran-SLM/Neo scores.
